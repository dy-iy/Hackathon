# risk_engine 小白版说明书

这份文档是为了让你在导师提问时能讲清楚：这个风控引擎到底在做什么、每个模块负责什么、为什么要这么拆、最后怎么串起来。

你可以先记住一句话：

> 这个系统不是让大模型直接判断风险，而是先把新闻/公告里的风险证据抽出来，再由 Python 规则和评分器确定风险分。

也就是说，LLM 在这里像“资料整理员”，负责从原文里找证据；真正的风险判断和打分，是后端代码完成的。

## 1. 整体在做什么

用户在前端聊天框输入一段加密货币相关文本，例如：

> 某交易所公告称，由于钱包维护，ETH 提现暂停，预计 2 小时后恢复，用户资金安全。

后端 `risk_engine` 会做这些事：

1. 先整理输入，把用户消息变成统一格式。
2. 用快速规则扫一遍，看文本里有没有“攻击、提现暂停、脱锚、监管、清算、诈骗、巨鲸转账”等风险信号。
3. 判断这条消息值不值得深入分析：
   - 如果几乎没风险，直接走 `fast_exit`，返回低风险。
   - 如果有风险信号，进入完整分析。
4. 选出可能相关的风险场景，例如“交易所异常风险”。
5. 给每个场景列一张“证据清单”，告诉 LLM 只需要抽哪些字段。
6. LLM 按证据清单从原文抽证据。
7. 每个风险场景都有自己的 Python 评分器，根据证据算出该场景的分数。
8. 把多个场景的评分结果合并。
9. 决策引擎根据合并后的结果算最终风险分。
10. 如果分数很高、证据冲突或置信度低，再进入校验节点。
11. 最后生成前端展示用的风险报告。

最终流程长这样：

```text
用户输入
  -> 输入标准化
  -> 快速风险信号扫描
  -> 自适应路由
       ├─ 弱信号：快速低风险报告
       └─ 有风险：完整案件分析
            -> 构建场景证据清单
            -> 抽取证据
            -> 多场景并行评分
            -> 合并评分结果
            -> 确定性决策打分
            -> 是否需要校验
                 ├─ 不需要：生成报告
                 └─ 需要：冲突校验 -> 重新应用决策 -> 生成报告
```

## 2. 文件怎么分工

`backend/app/risk_engine/` 下面现在按职责分成几类：

```text
risk_engine/
  graph.py                 # LangGraph 流程定义
  nodes.py                 # 节点编排，只负责串模块
  schemas.py               # Pydantic 数据结构
  core/                    # 路由、决策、校验、报告、汇总
  tools/                   # 输入整理、信号扫描、证据抽取
  prompts/                 # LLM prompt
  contracts/               # 证据字段合约
  scenario_evaluators/     # 各风险场景评分器
```

### `graph.py`：流程图

这个文件决定“先执行哪个节点，再执行哪个节点”。

它用 LangGraph 把整个风控流程串起来。你可以把它理解成导演，负责安排顺序：

```text
normalize_input
fast_signal_scan
adaptive_router
build_scenario_contracts
targeted_evidence_extractor
parallel_scenario_evaluators
merge_evaluation_results
deterministic_decision_engine
validation_gate
report_generator
```

如果导师问“你们的 multi-agent workflow 在哪里定义”，就说在 `graph.py`。虽然这里不是每个节点都叫 agent，但它们共同组成了一个多步骤的 agent 风控工作流。

### `nodes.py`：每个流程节点具体做什么

`graph.py` 只安排顺序，真正每一步怎么做，主要写在 `nodes.py`。

比如：

- `normalize_input_node`：整理用户输入。
- `fast_signal_scan_node`：快速扫风险信号。
- `adaptive_router_node`：决定快速退出还是完整分析。
- `fast_exit_report_node`：弱信号直接生成低风险报告。
- `build_scenario_contracts_node`：根据活跃场景生成证据清单。
- `targeted_evidence_extractor_node`：按清单抽证据。
- `parallel_scenario_evaluators_node`：并行跑各场景评分器。
- `merge_evaluation_results_node`：合并多个评分器结果。
- `deterministic_decision_engine_node`：算最终风险分。
- `validation_gate_node`：决定是否需要二次校验。
- `conflict_validator_node`：做冲突校验。
- `apply_validation_decision_node`：把校验建议应用到最终分数。
- `report_generator_node`：生成前端报告。

你可以把 `nodes.py` 理解成每个工位的操作说明。

### `schemas.py`：所有数据格式

这个文件定义系统内部传来传去的数据长什么样。

比如用户输入整理后，系统不想一直传一段普通字符串，而是传一个结构化对象，里面包括：

- 原始文本是什么
- 清洗后的文本是什么
- 语言是什么
- 输入类型是什么
- 识别出的实体有哪些

同理，风险信号、证据、评分结果、最终决策、校验建议，也都在 `schemas.py` 里定义了统一格式。

为什么要这么做？

因为这个系统是多节点流程，如果每个节点随便传字典，很容易字段写错、漏字段、格式混乱。用 Pydantic schema 可以让每一步输入输出更稳定。

### `tools/normalizer.py`：输入整理

这个模块负责把用户发来的文本变成系统能稳定处理的格式。

它会做几件事：

1. 去掉多余空格和换行。
2. 用文本内容生成一个 `case_id`，相当于这条风险案件的编号。
3. 判断语言，如果包含中文字符，就认为是中文。
4. 调用旧的聊天输入解析工具，提取一些基础信息，比如输入类型、实体、关键词引用。

比如用户输入：

```text
币安暂停 ETH 提现，预计 2 小时后恢复
```

整理后系统会知道：

- 原文是什么
- 清洗后的正文是什么
- 语言是中文
- 可能涉及交易所和资产实体

### `tools/signal_scanner.py`：快速风险信号扫描

这个模块是第一道风险筛查。

它不会做最终判断，只是快速回答：

> 这段文本里像不像有某类风险？

它会扫描这些风险方向：

- 攻击 / 漏洞利用
- 交易所异常
- 稳定币脱锚或储备问题
- 基础设施故障
- 监管执法
- 市场暴跌和清算
- 诈骗、钓鱼、跑路
- 巨鲸链上转账

它还会识别“降低风险”的表达，例如：

- 计划维护
- 已恢复
- 用户资金安全
- 无资金损失
- 只是监管讨论
- 只是安全研究
- 内部钱包调拨
- 未经证实传闻

这些降低风险的表达很重要。比如“暂停提现”听起来很严重，但如果同时写着“计划维护，2 小时后恢复，用户资金安全”，系统就不能直接给高风险。

所以 `tools/signal_scanner.py` 还会生成一些“最高分限制”信号，后面用于压分。

### `core/orchestrator.py`：自适应路由

这个模块决定这条文本走哪条大路径。

现在只有两条路径：

1. `fast_exit`
   - 文本风险信号很弱。
   - 不调用 LLM。
   - 不跑完整评分。
   - 直接返回低风险报告。

2. `deep_analysis`
   - 文本有风险信号。
   - 进入完整分析。
   - 后续会抽证据、跑评分器、生成完整报告。

它还会选出 `active_scenarios`，意思是“这次需要重点分析哪些风险场景”。

注意：它不会直接决定要不要进入校验。它只会给一个 `initial_validation_hint`，表示“我一开始看到了一些高风险或冲突迹象”。最终是否校验，由后面的 `validation_gate` 决定。

这样拆的好处是：

- 路由模块只管路径选择。
- 决策模块只管分数。
- 校验模块只管是否需要二次确认。

职责更清楚，导师问的时候也比较好解释。

### `core/scenario_router.py`：选择活跃风险场景

这个模块根据快速扫描结果，选出本次要分析的风险场景。

比如快速扫描发现：

- 交易所异常信号很强
- 监管信号比较弱
- 诈骗信号没有

那系统可能只选择：

- `S2_EXCHANGE_ABNORMALITY`
- `S0_GENERAL_UNKNOWN`

这里的 `S0_GENERAL_UNKNOWN` 是兜底场景，意思是“如果没有明确归类，也可以给一个综合风险判断”。

当前支持的场景有：

- `S0_GENERAL_UNKNOWN`：综合兜底
- `S1_ATTACK_EXPLOIT`：攻击 / 漏洞利用
- `S2_EXCHANGE_ABNORMALITY`：交易所异常
- `S3_STABLECOIN_RESERVE`：稳定币风险
- `S4_INFRASTRUCTURE_FAILURE`：基础设施故障
- `S5_REGULATORY_ENFORCEMENT`：监管执法
- `S6_MARKET_LIQUIDATION`：市场暴跌 / 清算
- `S7_FRAUD_GOVERNANCE`：诈骗 / 跑路 / 治理风险
- `S8_WHALE_ONCHAIN_FLOW`：巨鲸链上资金流

### `contracts/evidence.py`：证据清单

这个模块定义每个风险场景需要哪些证据字段。

可以把它理解成“审题清单”。

比如交易所异常场景需要问：

- 有没有暂停提现？
- 有没有暂停充值？
- 有没有暂停交易？
- 是不是计划维护？
- 有没有恢复时间？
- 有没有用户资金安全声明？
- 影响哪个交易所？
- 影响哪些资产？

攻击场景需要问：

- 攻击是否真的发生？
- 攻击方式是什么？
- 损失金额是多少？
- 用户资金是否受影响？
- 有没有攻击者地址？
- 是否已经修复？
- 是真实攻击，还是只是安全研究？

为什么要有 evidence contract？

因为不能让 LLM 自由发挥。我们只让它按清单抽证据，这样输出更稳定，也避免它直接编造风险结论。

### `tools/evidence_extractor.py`：证据抽取

这个模块负责从原文里抽证据。

它优先调用 LLM，但 LLM 的任务被严格限制：

- 只能根据原文回答。
- 只能填写证据字段。
- 不能给最终风险分。
- 不能给投资建议。
- 没有证据就填“缺失”。
- 只是传闻或无法确认，就填“不确定”。
- 如果说某个字段被确认，必须带原文证据片段。

举个例子，原文是：

```text
某交易所公告称，因钱包维护暂停 ETH 提现，预计 2 小时后恢复，用户资金安全。
```

它可能抽出：

- 暂停提现：确认
- 计划维护：确认
- 恢复时间：确认
- 用户资金安全声明：确认

如果 LLM 调用失败或返回格式不对，系统不会直接崩掉，而是用本地关键词和正则兜底。例如看到“暂停提现”就识别为提现暂停，看到“脱锚”就识别为稳定币脱锚，看到“1000 万美元”就尝试提取金额。

### `scenario_evaluators/`：各场景评分器

这个目录下面是各个风险场景自己的评分逻辑。

每个评分器只负责一个问题：

> 在我的这个场景里，根据已经抽到的证据，风险大概是多少？

比如：

#### `s1_attack.py`：攻击风险评分

它会看：

- 攻击是否发生
- 有没有损失金额
- 用户资金是否受影响
- 是否只是安全研究
- 是否已经修复

如果确认攻击且有大额损失，分数会高；如果只是安全研究，没有真实攻击，分数会被压低。

#### `s2_exchange.py`：交易所异常评分

它会看：

- 是否暂停提现
- 是否暂停交易
- 是否只是计划维护
- 是否有恢复时间
- 是否声明用户资金安全

如果确认暂停提现且没有恢复时间，风险较高；如果只是计划维护且有恢复时间，风险会被限制。

#### `s3_stablecoin.py`：稳定币风险评分

它会看：

- 是否提到脱锚
- 是否有储备问题
- 是否暂停赎回
- 发行方有没有说明

脱锚加储备问题或赎回暂停，风险会更高。

#### `s4_infra.py`：基础设施故障评分

它会看：

- 是否网络中断
- 是否停止出块
- 是否预言机故障
- 是否跨链桥暂停
- 用户交易是否受影响
- 资金是否有风险
- 是否已经恢复

如果已经恢复且没有资金风险，分数会被限制。

#### `s5_regulatory.py`：监管执法评分

它会看：

- 有没有监管机构
- 有没有被监管对象
- 有没有起诉、罚款、冻结、制裁等执法动作
- 是不是只是政策讨论
- 是不是监管利好

真正的执法动作风险更高；普通政策讨论不会给高分。

#### `s6_market.py`：市场清算评分

它会看：

- 价格跌幅是多少
- 清算金额是多少
- 是否有流动性压力
- 是不是只是行情分析或预测

真实暴跌和大额清算风险更高；普通行情评论会被压分。

#### `s7_fraud.py`：诈骗和治理风险评分

它会看：

- 是否钓鱼网站
- 是否假空投
- 是否冒充官方
- 是否假代币
- 是否诱导连接钱包
- 是否用户已经损失
- 是否跑路或 Rug Pull
- 是否资金被移走
- 是否只是普通团队变动

如果确认用户损失、跑路或资金被移走，风险会很高。

#### `s8_whale.py`：巨鲸链上流动评分

它会看：

- 是否大额转账
- 金额有多大
- 是否转入交易所
- 是否有卖压描述
- 是否只是内部钱包调拨

转入交易所并且文本指向卖压，风险会上升；内部调拨会被限制。

### `core/decision.py`：最终确定性打分

这个模块负责把各场景评分器的结果合成最终风险分。

它不是 LLM，是纯 Python 规则。

它会做这些事：

1. 从适用的场景里选主风险场景。
2. 优先选择具体风险场景，而不是兜底场景。
3. 以主场景分数作为基础分。
4. 如果还有其他高分副场景，给一点额外加分。
5. 应用评分器给出的最低分限制。
6. 应用评分器和快速扫描给出的最高分限制。
7. 如果有强证据，就不会轻易被弱的压分信号压掉。
8. 根据最终分数生成风险等级。

风险等级大致是：

- 91 分以上：极高风险
- 76 到 90：高风险
- 61 到 75：中高风险
- 41 到 60：中风险
- 21 到 40：轻微风险
- 0 到 20：低风险

这个模块很关键，因为它保证最终分数不是 LLM 随便说的，而是可解释的规则合成。

### `core/validator.py`：二次校验

这个模块负责处理“看起来可能有冲突”的情况。

比如：

- 文本里有“暂停提现”，但也有“计划维护、2 小时后恢复、资金安全”。
- 文本里有“漏洞”，但其实只是安全研究，没有真实攻击。
- 文本里有“大额转账”，但其实是官方内部钱包调拨。

这些情况如果不校验，系统可能误判高风险。

所以 `validation_gate` 会先判断要不要进校验。触发条件包括：

- 最终分数很高。
- 置信度较低。
- 有压分信号和高风险分数冲突。
- 路由阶段已经发现高危或冲突提示。

进入 `conflict_validator` 后，会按固定问题检查证据，然后给建议：

- `cap_score`：应该压低最高分。
- `raise_floor`：应该抬高最低分。
- `no_change`：不需要调整。

然后 `apply_validation_decision` 会把这个建议交回 `core/decision.py` 重新计算最终分数。

### `core/report.py`：生成前端报告

这个模块把内部复杂结果变成前端能展示的报告。

它会输出：

- 总结语
- 是否有风险
- 风险状态
- 最终风险分
- 风险等级
- 置信度
- 风险分类
- 命中的风险信号
- 降低风险的因素
- 证据片段
- 影响分析
- 处置建议
- 缺失信息
- 调分规则
- debug 信息

比如交易所计划维护的文本，报告可能会说：

- 这是交易所运维相关风险。
- 有暂停提现信号。
- 但也有计划维护、预计恢复、用户资金安全等缓和信息。
- 因此风险分不会很高。
- 建议继续关注官方公告和恢复进展。

## 3. 为什么不让 LLM 直接打分

导师很可能会问这个问题。

可以这样回答：

> 因为金融风控需要可解释和稳定。如果直接让 LLM 给风险分，结果可能不稳定，也很难解释为什么是这个分。我们把 LLM 限制在证据抽取环节，最终评分由 Python 规则完成，这样每个分数都有对应证据和规则来源。

这也是这个架构的核心亮点：

- LLM 负责理解文本和抽证据。
- Python 负责确定性评分。
- Validator 负责处理冲突和误报。
- Report generator 负责输出可读结论。

## 4. 一条样例怎么走完整流程

输入：

```text
某交易所公告称，因钱包维护暂停 ETH 提现，预计 2 小时后恢复，用户资金安全。
```

系统处理过程：

1. `tools/normalizer.py` 整理文本，识别语言和基础实体。
2. `tools/signal_scanner.py` 发现“暂停提现”，认为可能是交易所异常。
3. 同时也发现“钱包维护”“预计恢复”“用户资金安全”，这些是缓和信号。
4. `core/orchestrator.py` 判断这不是完全无风险，进入完整分析。
5. `core/scenario_router.py` 选择 `S2_EXCHANGE_ABNORMALITY` 交易所异常场景。
6. `contracts/evidence.py` 告诉抽取器要找：是否暂停提现、是否计划维护、恢复时间、资金安全声明。
7. `tools/evidence_extractor.py` 从原文抽到这些证据。
8. `s2_exchange.py` 开始评分：
   - 暂停提现会提高风险。
   - 计划维护、恢复时间、资金安全声明会降低风险。
9. `merge_evaluation_results` 合并评分结果。
10. `core/decision.py` 算最终分，应用压分规则。
11. `validation_gate` 判断是否需要二次校验。
12. `core/report.py` 输出报告。

最终结论大概率不是高风险，而是：

> 有交易所运维风险信号，但由于文本说明是计划维护、有恢复时间且用户资金安全，风险被限制，建议持续关注恢复进展。

## 5. 导师提问时可以怎么讲

### 问：你们这个系统的核心创新是什么？

答：

我们没有让大模型直接给风险分，而是采用 evidence contract 的方式，让大模型只抽证据。风险评分、压分、抬分和最终决策都由 Python 规则完成。这样结果更稳定，也更容易解释。

### 问：为什么要分这么多节点？

答：

因为每个节点职责不同。输入标准化、信号扫描、场景选择、证据抽取、场景评分、结果合并、最终决策、冲突校验和报告生成分别拆开后，每一步都能单独解释、单独调试，也方便后续加新场景。

### 问：如果 LLM 抽错了怎么办？

答：

第一，prompt 限制它只能按原文抽字段，不能自由生成结论。第二，Pydantic 会校验返回结构。第三，如果 LLM 失败或格式不对，本地启发式规则会兜底。第四，最终评分不是 LLM 做的，所以风险分不会完全依赖 LLM。

### 问：误报怎么处理？

答：

系统会识别缓和信号，比如计划维护、已恢复、用户资金安全、安全研究、内部调拨等。这些信号会变成 cap 规则限制最高分。高风险或冲突情况下还会进入 validator，再做一次固定问题校验。

### 问：怎么新增一个风险场景？

答：

一般要加这些地方：

1. 在 `schemas.py` 加新的场景 ID。
2. 在 `core/scenario_router.py` 加场景说明。
3. 在 `contracts/evidence.py` 定义这个场景需要哪些证据字段。
4. 在 `tools/evidence_extractor.py` 加兜底抽取规则。
5. 在 `scenario_evaluators/` 新增对应评分器。
6. 在 `scenario_evaluators/base.py` 注册评分器。
7. 在 `core/report.py` 加前端展示用的风险分类名称。

## 6. 最后一句话总结

这个 `risk_engine` 的设计思路是：

> 先用规则快速发现风险方向，再让 LLM 按证据清单抽取原文证据，随后由多个 Python 场景评分器独立评分，合并后交给确定性决策引擎生成最终风险分，必要时再做冲突校验，最后输出可解释的风控报告。
