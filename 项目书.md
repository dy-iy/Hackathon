# Codex 项目书：CryptoRisk Evidence-Contracted Case Engine 重构方案

你现在是本项目的代码重构助手。请严格按照本项目书执行，不要自由发挥成另一个复杂 multi-agent 系统。

## 一、项目背景

本项目是一个面向黑客松金融风控赛道的加密货币风险分析系统。

现有系统形态：

* 前端：Next.js / React / TypeScript / Tailwind CSS
* 后端：FastAPI / Python / Pydantic
* Agent：LangChain / LangGraph
* 模型 API：DeepSeek API
* 产品形态：前端聊天窗口 + 后端风险分析服务

系统目标是对以下内容进行风险识别、风险分类、风险评分、证据抽取、影响分析和处置建议生成：

* 加密货币新闻
* 交易所公告
* 项目方事件
* 监管执法信息
* 链上安全事件
* 稳定币异常
* 市场暴跌 / 清算事件
* 巨鲸异动
* 交易所提现 / 充值 / 交易异常

当前旧系统存在问题：

1. Agent 数量过多，链路复杂。
2. 多个 LLM Agent 反复对同一文本做主观判断。
3. Agent 之间上下文污染，前序结论容易带偏后续结论。
4. 风险分误判明显，尤其是关键词误伤、监管讨论误判、计划维护误判、安全研究误判。
5. 分析速度慢。
6. 旧规则脚本有价值，但不能继续直接作为最终风险裁判。
7. 现有前端接口需要尽量保持稳定，不能因为后端重构导致前端大面积改动。

本次重构的目标不是继续增加 Agent，而是把系统重构为：

```text
CryptoRisk Evidence-Contracted Case Engine
证据契约驱动的加密资产风险案件引擎
```

核心原则：

```text
以风险案件为核心
以证据契约约束 LLM
以场景评分卡做裁决
以问题式验证降低误判
以评测集持续校准
```

一句话设计原则：

```text
LLM 不直接判最终风险分，只负责按证据契约抽取结构化证据；
风险分、风险等级、风险状态由 Python 评分卡和确定性决策引擎生成。
```

## 二、最高优先级约束

以下约束优先级最高，必须遵守。

### 1. 第一轮只读代码，不允许直接修改

第一步请只做项目审计，不要修改任何文件。

你需要先阅读当前项目结构，输出：

* 当前 `/api/chat` 或相关聊天接口位置
* 当前 Agent workflow 入口
* 当前 LangGraph graph 定义位置
* 当前规则评分脚本或规则逻辑位置
* 当前报告生成位置
* 当前前端调用后端 API 的位置
* 当前返回给前端的数据结构
* 哪些模块可以复用
* 哪些模块应该废弃
* v6 重构会影响哪些文件
* 推荐的最小可行改造路径

只有在用户确认后，才能开始写代码。

### 2. 不允许破坏现有前端 API

前端现有请求方式必须尽量保持兼容。

不要在前端写死：

```text
localhost
127.0.0.1
8001
```

不要让前端因为后端重构而大面积改动。

如果需要新增字段，请优先放在：

```json
{
  "v6_result": {},
  "debug": {}
}
```

不要随意删除旧字段。

### 3. 旧 pipeline 必须保留 fallback

在 v6 smoke eval 通过前，保留旧 pipeline fallback。

推荐方式：

```text
USE_V6_RISK_ENGINE=true / false
```

或者类似环境变量。

如果 v6 pipeline 出错，应能回退到旧逻辑，避免接口直接崩溃。

### 4. 所有新模块必须无 import 副作用

禁止在 import 时执行：

* 读取 CSV
* 写文件
* 调用 LLM
* 请求网络
* 启动爬虫
* 初始化大任务
* 自动运行评测

模块 import 只能定义函数、类、常量。

### 5. LLM 输出必须经过 Pydantic schema 校验

DeepSeek / LLM 输出不能直接进入决策引擎。

必须经过：

```text
Pydantic schema parse
字段合法性校验
缺失字段 fallback
异常捕获
```

如果 LLM 输出 JSON 解析失败，必须有 fallback：

```text
risk_status = insufficient_evidence
confidence 降低
不允许直接报错炸穿接口
```

### 6. LLM 不能直接决定最终 risk_score

LLM 只负责：

* 填 evidence contract
* 抽取 evidence_text
* 判断字段状态 confirmed / denied / missing / uncertain
* 标记缺失信息
* 必要时回答固定验证问题

LLM 不允许直接输出最终风险分作为最终结论。

最终风险分必须由：

```text
Scenario Evaluators
Decision Policy Engine
```

生成。

### 7. Conflict Validator 不能直接改分

`Question-based Conflict Validator` 只能输出：

```json
{
  "action": "cap_score / raise_floor / no_change",
  "score_cap": 45,
  "score_floor": null,
  "reason": "..."
}
```

最终是否应用 cap/floor，必须回到 `Decision Policy Engine` 执行。

正确流程是：

```text
Decision Policy Engine
  ↓
Need validation?
  ├─ No → Report Generator
  └─ Yes
       ↓
     Question-based Conflict Validator
       ↓
     Decision Policy Engine apply validation
       ↓
     Report Generator
```

### 8. Evaluation Harness 是离线脚本

评测系统不进入实时 `/api/chat` 请求链路。

生产请求链路：

```text
Input → ... → Report Generator
```

离线评测链路：

```text
eval_dataset → v6 pipeline → metrics report
```

### 9. 场景 Evaluator 使用 Python 评分卡

场景 evaluator 不调用 LLM。

每个场景使用确定性 Python 评分卡。

### 10. 第一阶段不要一次性做满所有场景

第一阶段优先实现：

```text
S0 General Risk / Unknown
S1 Attack / Exploit
S2 Exchange Abnormality
S3 Stablecoin / Reserve minimal stub
S5 Regulatory Enforcement
S6 Market / Liquidation
```

第二阶段再扩展：

```text
S4 Infrastructure Failure
S7 Fraud / Governance
S8 Whale / On-chain Flow
```

## 四、核心模块职责

### 1. Input Normalizer

职责：

* 统一用户输入、新闻文本、公告文本、URL 摘要等输入格式。
* 合并标题和正文。
* 清洗空文本、HTML、异常字符。
* 提取基础 source hint、时间 hint、语言 hint。
* 输出标准 `RiskCaseInput`。

输出示例：

```json
{
  "case_id": "hash_xxx",
  "raw_text": "...",
  "title": "...",
  "content": "...",
  "source_url": "...",
  "source_name": "...",
  "published_at": null,
  "language": "zh",
  "metadata": {}
}
```

### 2. Fast Signal Scan

职责：

* 迁移旧规则脚本的有用部分。
* 仅作为信号扫描层。
* 不直接决定最终风险分。
* 不直接决定最终风险等级。

输出三类信号：

```json
{
  "positive_signals": [],
  "negative_signals": [],
  "cap_signals": []
}
```

正向信号示例：

```json
{
  "type": "withdrawal_suspension",
  "matched_terms": ["暂停提现"],
  "strength": 0.72,
  "scenario_hint": "S2_EXCHANGE_ABNORMALITY"
}
```

负向信号示例：

```json
{
  "type": "planned_maintenance",
  "matched_terms": ["计划内维护"],
  "strength": -0.25
}
```

封顶信号示例：

```json
{
  "type": "rumor_without_confirmation",
  "score_cap": 55,
  "reason": "仅社媒传闻，缺少官方或链上证据"
}
```

注意：

旧规则脚本里原有的 `risk`、`rule_label`、`rule_primary_type` 可以作为 debug 输出保留，但不能作为最终决策来源。

---

### 3. Adaptive Orchestrator

职责：

根据 `Fast Signal Scan` 的结果决定走哪条路径。

路径 A：Fast Exit

条件示例：

```text
规则信号弱
无高危实体
文章类型类似普通行情日报 / 普通项目公告 / 普通观点文章
没有攻击、提现异常、监管执法、稳定币脱锚、清算等强信号
```

输出：

```json
{
  "path": "fast_exit",
  "risk_status": "low_risk",
  "risk_score": 10,
  "risk_level": "low",
  "reason_codes": ["weak_rule_signal", "no_high_risk_scenario_detected"]
}
```

Fast Exit 也必须有解释，不能只返回“低风险”。

路径 B：Standard Analysis

条件示例：

```text
存在 1-2 个明确候选场景
风险信号中等
没有明显冲突
```

执行：

```text
router → contract_builder → evidence_extractor → scenario_evaluators → decision_engine
```

路径 C：High-risk Validation Path

条件示例：

```text
高危场景
规则信号高但文章类型可疑
规则信号与证据冲突
final_score >= 75
confidence < 0.6
```

执行：

```text
standard analysis + conflict validator
```

* 默认 Top-2。
* 复杂情况 Top-3 / Top-4。
* 低风险可以 Top-1 或 fast exit。
* 必须支持兜底场景 S0。

场景枚举：

```text
S1_ATTACK_EXPLOIT
S2_EXCHANGE_ABNORMALITY
S3_STABLECOIN_RESERVE

S5_REGULATORY_ENFORCEMENT
S6_MARKET_LIQUIDATION
```

第一阶段实现重点：

```text

S2_EXCHANGE_ABNORMALITY
S3_STABLECOIN_RESERVE minimal stub
```

### 6. Targeted Evidence Extraction Agent

职责：

这是唯一主要调用 DeepSeek 的分析模块。

它不能问：

```text
请分析这条新闻风险高不高
```

必须问：

```text
请只根据原文，为以下 evidence contract 填字段。
没有证据就填 missing。
不要推断。
每个字段必须绑定 evidence_text。
```

字段统一格式：

```json
{
  "field": "withdrawal_suspended",
  "value": true,
  "status": "confirmed",
  "evidence_text": "...",
  "confidence": 0.86
}
```

`status` 只能取：

```text
confirmed
denied
missing
uncertain
not_applicable
```

规则：

1. 如果原文没有证据，必须填 `missing`。
2. 如果原文明确否认，必须填 `denied`。
3. 如果只是传闻或预测，不能填 `confirmed`。
4. 每个 confirmed 字段必须有 evidence_text。
5. 不允许输出最终 risk_score。
6. 不允许输出最终 risk_level。
7. 不允许做长篇自由分析。

LLM 输出必须经过 Pydantic schema 校验。

---

### 7. Parallel Scenario Evaluators

职责：

每个候选场景一个 Python 评分卡。

不同场景可以异步并行执行，但必须满足：

1. 每个 evaluator 只读取同一份 `RiskCaseState`。
2. 不允许 evaluator 互相读取彼此结论。
3. 所有 evaluator 输出统一 schema。
4. evaluator 不调用 LLM。

统一输出：

```json
{
  "scenario": "S2_EXCHANGE_ABNORMALITY",
  "is_applicable": true,
  "scenario_score": 68,
  "confidence": 0.82,
  "severity": "medium_high",
  "score_cap": 45,
  "score_floor": null,
  "reason_codes": [
    "withdrawal_suspended",
    "official_notice",
    "planned_maintenance"
  ],
  "missing_evidence": [
    "fund_safety_statement"
  ]
}
```

## 六、Decision Policy Engine

职责：

最终裁决层，完全由 Python 执行。

输入：

```text
signal_scan_result
evidence_items
scenario_evaluations
validation_suggestion optional
```

输出：

```json
{
  "risk_score": 72,
  "risk_level": "high",
  "risk_status": "confirmed_risk",
  "primary_scenario": "S2_EXCHANGE_ABNORMALITY",
  "secondary_scenarios": [],
  "confidence": 0.81,
  "reason_codes": [],
  "score_caps_applied": [],
  "score_floors_applied": []
}
```

风险状态枚举：

```text
low_risk
potential_risk
confirmed_risk
insufficient_evidence
resolved_or_mitigated
false_positive_suppressed
```

分数规则：

```text
主场景分数 = 最高可信场景分
次级风险加成 = 0-10
证据不足降低 confidence
传闻 / 预测 / 历史复盘触发 score cap
确认攻击 / 明确损失触发 score floor
最终分数限制在 0-100
```

风险等级建议：

```text
0-20: low
21-40: low_medium
41-60: medium
61-75: medium_high
76-90: high
91-100: critical
```

证据不足必须是合法输出，不是异常：

```json
{
  "risk_status": "insufficient_evidence",
  "risk_score": 42,
  "confidence": 0.38,
  "reason": "仅有传闻或弱信号，缺少官方公告、链上证据或明确事件事实"
}
```

## 七、Question-based Conflict Validator

职责：

只在必要时运行。

触发条件：

```text
final_score >= 75
confidence < 0.6
规则信号高但 evidence quality 弱
规则信号低但 evidence 显示严重事件
候选场景分数接近
文章类型是安全研究 / 历史复盘 / 观点预测，但系统给了高分
```

Validator 不能自由复核，不能重新写风险报告，不能直接修改最终分。

它只能回答固定验证问题，并输出 cap/floor 建议。

示例问题：

S1 Attack：

```text
文本是否明确说明攻击已经发生？
是否明确说明资金损失？
是否只是安全研究、漏洞披露或审计报告？
是否说明漏洞已修复？
是否说明用户资金未受影响？
当前高分是否缺少必要证据支撑？
```

S2 Exchange：

```text
是否明确暂停提现 / 充值 / 交易？
是否只是计划维护？
是否有明确恢复时间？
是否说明用户资金安全？
是否已经恢复？
是否只是用户传闻？
```

S5 Regulatory：

```text
是否有明确监管主体？
是否有明确执法动作？
是否只是监管讨论、草案、咨询或观点？
是否存在驳回、放行、豁免等利好信息？
```

输出示例：

```json
{
  "action": "cap_score",
  "score_cap": 45,
  "score_floor": null,
  "reason": "文本为安全审计披露，未说明攻击已经发生，也没有资金损失"
}
```

或者：

```json
{
  "action": "raise_floor",
  "score_cap": null,
  "score_floor": 80,
  "reason": "文本明确提到未授权铸造、攻击者地址和资金损失"
}
```

最终应用由 `Decision Policy Engine apply validation` 执行。

## 十二、验收标准

第一阶段重构完成后，必须满足：

1. 后端服务能正常启动。
2. `/api/chat` 或现有聊天接口不崩。
3. 前端聊天窗口能正常收到结果。
4. 普通低风险新闻能 fast exit。
5. 安全审计 / 漏洞披露但未攻击，不应直接判极高风险。
6. 计划内维护 + 有恢复时间，不应判极高风险。
7. 监管讨论 / 草案 / 咨询，不应判高风险执法。
8. 明确攻击 + 明确损失，应能判高风险。
9. 明确暂停提现且无恢复时间，应能判中高或高风险。
10. LLM JSON 解析失败时系统不崩，应返回 insufficient_evidence 或 fallback。
11. Evaluation Harness 能跑 smoke eval。
12. 旧 pipeline fallback 可用。
